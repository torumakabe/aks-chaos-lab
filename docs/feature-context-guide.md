# フィーチャーコンテキストドキュメント ガイド

フィーチャーコンテキストドキュメントは、複数セッションにまたがる開発での意思決定を記録する軽量な作業記録です。
正式なテンプレートではありません。必要な内容に合わせて自由に書いてください。

## 作成するタイミング

- 複数セッションにまたがる機能開発
- 複数の開発者が同じ機能に取り組む場合
- 1セッションで完了する作業や軽微な修正では不要

## 含めるべき内容

例（通知サービス機能の場合）:

```markdown
# Feature: Notification Service v1

## 決定事項
| 決定                          | 理由                                  | 却下した代替案                                   |
|-------------------------------|---------------------------------------|-------------------------------------------------|
| BullMQ を直接使用（ラッパーなし） | ネイティブのリトライ + バックオフで十分    | RetryQueue 抽象化（不要な間接層）                  |
| 関数ベースのサービス             | コードベースの規約に合わせる              | クラスベース（規約に反するため却下）                 |

## 制約
- v1 はメールのみ（SMS/プッシュ通知なし）
- 既存の認証ミドルウェアを使用すること

## 未解決の課題
- [ ] レート制限の戦略（プロダクト側の入力待ち）

## 進捗
- [x] 設計承認済み
- [x] NotificationHandler 実装済み
- [ ] DeliveryTracker（次のセッションで対応）
```

## ライフサイクル

1. 複数セッションにまたがる作業を開始する際に `docs/features/<feature-name>.md` を作成
2. 自然な区切り（重要な決定の後など）で更新する。儀式的にではなく
3. 機能が完成したら、重要な決定を `docs/design.md` 等に昇格させ、このファイルは削除する
